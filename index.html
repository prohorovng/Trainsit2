3. Key journeys

Journeys after the blocker answers.

Journey 1 — New visitor → quote → app handoff

  • User lands on the homepage.
  • User understands the proposition and clicks Get a Quote.
  • User enters route details on the website.
  • Website calls the quote API and renders loading + result states.
  • User clicks Book on a chosen quote option.
  • User is redirected to https://app.trainsit.com/login?option_id={option_id}.
  • App handles login / registration and then continues the flow.

Journey 2 — Evaluator → validation → contact

  • User reviews homepage, Solutions, About, and FAQ.
  • User validates scope, supported routes, operational logic, and trust indicators.
  • User either contacts the team or moves into Get a Quote.

Journey 3 — Existing user → login → dashboard

  • User clicks Login or Dashboard on the website.
  • User is routed to the app.
  • App already handles logged-in vs not-logged-in logic.

5. Technical architecture

Updated architecture after the answers.

Website

Webflow-based marketing layer on trainsit.com with content, SEO, analytics, FAQ, contact, and Get a Quote UI.

Public quote integration

The website sends requests to the public quote endpoint and renders success, loading, no-results, and error states directly on the site.

App handoff

Booking and authenticated continuation always move to app.trainsit.com. The current confirmed Book handoff is /login?option_id={option_id}.

Technical constraint Discovery impact
Quote API exists and is documented Architecture no longer needs discovery around whether Get a Quote is feasible. It is now an implementation design topic.
Public search is open No website auth is required for quote search, but abuse mitigation and traffic control remain relevant.
API latency is 3–15 seconds Loading UX, progress messaging, and timeout behavior become mandatory discovery items.
No-route fallback should become “Request This Route” No-results state is no longer just an error state. It becomes a separate recovery flow.
App already handles dashboard auth redirect Discovery should stop discussing generic auth routing and instead validate specific Book and option handoff behavior.
6. Dependencies and required validations

What still must be checked before build starts.

Technical validations

  • Validate the real quote API request/response cycle using actual frontend calls.
  • Confirm exact request payload and response mapping for result cards.
  • Validate how option_id behaves through login and whether the user lands on the right post-login state.
  • Confirm no-results payload shape for Request This Route logic.
  • Validate rate-limit behavior under public traffic.

Content and release dependencies

  • Solutions, FAQ, insurance, coverage, and operations content still need to be supplied or written.
  • Logo and final visual direction are still dependencies for design quality, even if platform choice is fixed.
  • Staging is not required immediately, but release governance is still a future risk if no staging process exists.
